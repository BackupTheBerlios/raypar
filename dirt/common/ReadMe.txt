Это файл для обмена мнениями по общему оформлению и организации проекта. 
Вещи касающиеся сервера и клиента давайте писать в ServerReadMe.txt и
ClientReadMe.txt

Структуру каталогов проекта я вижу следующим образом

\\Root Project Folder
   \\CLIENT   - файлы клиента (DSP, исходный код, картинки и 
				всякий мусор, который появляется при сборке, 
				короче всё, что не будет явно отнесено в другие папки 
				и имеет отношение к клиенту)
   \\SERVER   - файлы сервера (то же, что и для клиента)
   \\COMMON   - исходные код, который включается и в сервер и в клиент
				( сюда относится протокол связи, а что ещё пока не вижу)
   \\BUILT    - сюда кладутся исполняемые файлы, собранные в Debug режиме
				( что бы не надо было лазить сначала в одну папку за сервером, 
				а потом в другую за клиентом)
   \\DOCS	  - всяческая документация по проекту, в частности этот файл
      
	то, чего пока нет, но видимо появится

	\\HELP    - файлы, нужные для сборки справки (правда м.б. нужно будет делать
				справку и к серверу и к клиенту - тогда будут подпапки в \\CLIENT и
				\\SERVER)

	\\RELEASE - всё собранное под RELEASE
	\\DISTRIB - дистрибутив, когда (если) он всё-таки появится.

В общем как-то много получилось, но вроде бы всё нужно.

Теперь пару слов по-поводу оформления кода (как я понимаю, это один из ОСНОВНЫХ критериев,
  по которому Тормасов оценивает проги - он в этом вопросе не собаку съел, 
  а кита, поэтому халявы не будет ;( )

Предлагаю следующее:

1) Делать шапки файлов по моему образцу, т.е. вида:

//****************************************
//** FileName.ext **
// Created By: KIRILL
// On :10/27/2003 01:49:50
// Comments: зачем создан файл
//
//***********************************
// REVISION by ..., on ...
// Comments: ...
//***********************************

2) При внесении исправлений, особенно в ЧУЖОЙ файл тщательно описывать где, что 
и зачем поменяли. Предлагаю это делать в формате:
//***********************************
// REVISION by KIRILL, on 10/27/2003 01:49:50
// Comments: в SuperFunction() добавил проверку входных параметров
//***********************************

и при этом новые ревижены писать ПОД старыми.


3) ПИСАТЬ КОММЕНТАРИИ !!! ОБЯЗАТЕЛЬНО !!!
Тормасов: "Комментарии должны быть в среднем один на три строчки"


4) Тормасов считает, что программы без ASSERT() работать не могут в принципе. 
IMHO он прав:) 


5) НЕ удалять файлы! Если файл вам больше не нужен (перенесли код в другое место 
или решили переименовать файл), то напишите в самом верху файла следующее:

#error Not Used !!!
// KIRILL, 10/27/2003 01:49:50
// Всё перенёс в BlaBla.cpp

но не стирайте файл! Если файл стереть, то люди потом начинают его искать и спрашивать
" а почему ты мне не прислал DeletedFile.cpp??? Пришли, а то у меня ничего не собирается!"

6) ПИСАТЬ КОММЕНТАРИИ!!! (Специально повторяюсь!)

7) В общем предлагаю подумать и либо со мной согласиться и делать так, как я 
описал, либо не согласиться и предложить свои идеи, главное, чтобы все оформляли
код одинаков, а не пестро и вразнобой.
